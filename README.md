# GitOps Test — Flux + SOPS + MariaDB Operator

A GitOps reference setup using Flux CD with SOPS-encrypted secrets and a MariaDB operator-driven database on a Virtuozzo Kubernetes cluster.

## Architecture

```
Git Repository
│
├── clusters/virtuozzo/              ← Flux reconciliation entrypoint
│   ├── flux-system/                  ← Generated by flux bootstrap (controllers + source)
│   ├── infrastructure-kustomization  ← Flux Kustomization → infrastructure/
│   └── apps-kustomization            ← Flux Kustomization → apps/ (SOPS decryption enabled)
│
├── infrastructure/
│   └── mariadb-operator/             ← HelmRepository + HelmRelease
│
├── apps/
│   └── my-app/                       ← Full application stack
│       ├── MariaDB CR               ← MariaDB instance (v11.4.2)
│       ├── Database CR              ← Database "my-app-data"
│       ├── User CR                  ← Database user with password from secret
│       ├── Grant CR                 ← Privileges on the database
│       ├── ConfigMap                ← App configuration (DB host, port, etc.)
│       ├── Deployment               ← Stateless app (2 replicas)
│       ├── Service                  ← ClusterIP service on port 8080
│       └── secrets/                 ← SOPS-encrypted db password
│
├── sample-app/                       ← Reference Python Flask API
│   ├── app.py                        ← REST API (CRUD) with MariaDB connection
│   ├── requirements.txt
│   └── Dockerfile
│
├── scripts/                          ← End-to-end automation
│   ├── 01-install-tools.sh           ← Install kubectl, helm, flux, sops, age
│   ├── 02-bootstrap-flux.sh          ← Bootstrap Flux on the cluster
│   ├── 03-configure-sops.sh          ← Generate age keys, create K8s secret, encrypt secrets
│   ├── 04-deploy-infrastructure.sh   ← Deploy MariaDB operator via Flux
│   └── 05-deploy-app.sh             ← Build image, encrypt secrets, deploy app
│
└── .sops.yaml                        ← Root SOPS encryption config
```

## Prerequisites

- A Virtuozzo Kubernetes cluster (kubeconfig configured)
- A GitHub personal access token with repo permissions
- Docker (optional, for building the sample app image)

## Quick Start

```bash
# 1. Install CLI tools
./scripts/01-install-tools.sh

# 2. Bootstrap Flux on the cluster
export GIT_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx
./scripts/02-bootstrap-flux.sh

# 3. Set up SOPS encryption
./scripts/03-configure-sops.sh

# 4. Deploy MariaDB operator
./scripts/04-deploy-infrastructure.sh

# 5. Deploy the sample application
./scripts/05-deploy-app.sh
```

After deploying, port-forward and test the API:

```bash
kubectl port-forward -n my-app svc/my-app 8080:8080
curl http://localhost:8080/health           # Health check + DB connection
curl http://localhost:8080/api/items         # List items
curl -X POST http://localhost:8080/api/items \
  -H "Content-Type: application/json" \
  -d '{"name": "test-item"}'                # Create item
```

## Component Details

### Flux CD

Flux is bootstrapped with the `flux bootstrap git` command, targeting the `clusters/virtuozzo/` path. The bootstrap creates:

- All Flux controllers in the `flux-system` namespace
- A `GitRepository` source pointing to this repo
- The local `clusters/virtuozzo/flux-system/` directory

Two Flux `Kustomization` resources (defined in `clusters/virtuozzo/`) tell Flux what to reconcile:

| Kustomization | Path | Depends On | SOPS |
|---|---|---|---|
| `infrastructure` | `./infrastructure/` | — | No |
| `apps` | `./apps/` | `infrastructure` | Yes |

### MariaDB Operator

The operator is installed via a HelmRelease from `mariadb-operator/mariadb-operator` (Helm chart). Once running, the following CRDs become available:

| CR | Purpose | Example |
|---|---|---|
| `MariaDB` | MariaDB server instance | `my-app-db` |
| `Database` | Logical database | `my-app-data` |
| `User` | Database user | `my-app-user` |
| `Grant` | User privileges | SELECT, INSERT, etc. |
| `Connection` | Connection string helper | — |

### SOPS + Age

Secrets are encrypted with SOPS using age keys. The flow:

1. `age-keygen` generates a key pair
2. The private key is stored as a Kubernetes secret (`sops-age`) in `flux-system`
3. The `apps` Kustomization references this secret for SOPS decryption
4. `password.txt` is encrypted into `db-secret.enc.yaml`
5. Flux decrypts and applies the secret during reconciliation

To add or update an encrypted secret:

```bash
# Create/update the plaintext
echo "new-password" > apps/my-app/secrets/password.txt

# Re-encrypt
kubectl create secret generic db-secret \
  --namespace=my-app \
  --from-file=password=apps/my-app/secrets/password.txt \
  --dry-run=client -o yaml \
  | sops --encrypt --input-type=yaml --output-type=yaml \
    --encrypted-regex='^(data|stringData)$' \
    - > apps/my-app/secrets/db-secret.enc.yaml
```

### Sample Application

The `sample-app/` directory contains a Python Flask REST API that:

- Connects to MariaDB with connection pooling
- Creates an `items` table on startup
- Exposes `GET /health`, `GET /api/items`, `POST /api/items`
- Reads DB configuration from environment variables (ConfigMap + Secret)

Build and push:

```bash
docker build -t my-app:latest sample-app/
docker tag my-app:latest <registry>/my-app:latest
docker push <registry>/my-app:latest
```

Then update `apps/my-app/deployment.yaml` with the image reference.
